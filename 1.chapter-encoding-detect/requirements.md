# 章节文件编码自适应 — 需求规格

## 概述

导入章节文件时自动识别编码（UTF-8 优先，GBK/UTF-16 兜底），让 GBK 等非 UTF-8 编码的 txt 不再整章乱码。

## 项目信息

- 项目名: fanqie-uploader
- 架构类型: Chrome MV3 浏览器扩展（零依赖，无构建；改动集中在 popup 层）

## 需求版本

| 日期 | 版本 | 说明 |
| ---- | ---- | ---- |
| 2026-07-19 | v1 | 初始需求（对标竞品「番茄小说注入神器」UTF-8/GBK 双编码） |

## 用户故事

- 作为用旧编辑器/记事本导出 GBK 稿的网文作者，我想要导入 GBK 编码的 txt 时中文不乱码，以便直接用现成稿子而不必先转码。
- 作为 Windows「Unicode 另存」用户，我想要 UTF-16 文件也能正确读入，以便不挑编码来源。

## 功能需求

1. [F-001] 读取章节文件时不再固定按 UTF-8 解码，改为「探测 + 解码」：BOM 优先，无 BOM 则 UTF-8 严格解码，失败回退 GBK（GB18030 解码器）。
2. [F-002] 识别并正确解码带 BOM 的 UTF-8 / UTF-16LE / UTF-16BE 文件。
3. [F-003] 解码逻辑为顶层具名纯函数，可被 tests/logic.test.cjs 源码抽取执行。
4. [F-004] 解码后文本继续走现有 splitTitleBody / numFromName / numFromText 流水线，下游零改动、无回归。

## 非功能需求

- 性能: 单文件解码为内存操作，导入几十章无感知延迟。
- 安全: 不新增任何 manifest 权限；文件仅在浏览器内存解码，不落盘、不上传（沿用现有承诺）。
- 兼容性: 仅用浏览器原生 `TextDecoder`（含 gbk/gb18030 标签，Chrome 原生支持），零第三方依赖。

## 验收标准

- [ ] [AC-001] 一份 GBK 编码、含中文标题与正文的 txt，导入后标题/正文/字数正确，无 `�`。
- [ ] [AC-002] UTF-8（含/不含 BOM）文件行为与改动前完全一致，无回归。
- [ ] [AC-003] UTF-16LE（Windows 记事本「Unicode」另存）文件正确读入。
- [ ] [AC-004] 空文件、或既非合法 UTF-8 又非合法 GBK 的文件：不崩溃；GBK 解码器（非 fatal）会把非法字节替换为 `�`，产出的乱码章由 minWords 兜底过滤——不追求「解码异常抛错」路径（TextDecoder('gbk') 非 fatal 永不抛）。
- [ ] [AC-005] `bash tests/run.sh` 逻辑测试全绿（含新增编码用例）。

## 依赖

- 浏览器原生 `TextDecoder`（'utf-8' fatal 模式 / 'gbk' / 'utf-16le' / 'utf-16be'）。无外部库。

## 开放问题

- Q1（已决，本期不做）：导入后「首章预览」让用户肉眼确认编码/切分——按 PRD 建议先不做，观察反馈后再定。
- 已知边界：极少数 GBK 文本恰为合法 UTF-8 字节序列时会被判为 UTF-8。`fatal:true` 严格模式可拦住绝大多数；作为已知边界记录，Q1 预览可作为后续兜底。
