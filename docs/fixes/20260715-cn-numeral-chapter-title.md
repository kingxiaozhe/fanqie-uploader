# 中文数字章节名：标题框被填入"第X章"前缀（章节号已单独有框）

- 日期：2026-07-15　状态：已修复
- 修前证据：用户运行日志（新书批次）`填标题：章节号框✓ 标题="第二章 重算这笔账"`、
  `标题="第十一章 茶楼机锋"` ——中文数字前缀原样进标题框；同日志阿拉伯数字章节
  `第 56 章 代价` 被正确剥为 `代价`。用户报告现象一致（例子"第20章 你好啊林光源"，
  实际失败面为中文数字写法；阿拉伯写法经测试本来就正确）。

## 现象

md/txt 章节名（文件名或正文首行）为"第二十章 你好啊林光源"时，番茄标题框被填入
"第二十章 你好啊林光源"；期望只填"你好啊林光源"（章节号有独立输入框）。

## 复现

`node tests/logic.test.cjs` 修复前：`pureTitle` 不存在（旧逻辑内联在 fillTitle），
剥前缀正则 `/^\s*第\s*\d+\s*章…/` 对中文数字不命中；`numFromName("第二十章 xx.md")`
返回 null（文件排序也因此失效，全部按 0 排）。

## 根因

「第N章」识别只认阿拉伯数字（`\d+`），散布五处：publisher 剥标题前缀 / 章节号兜底、
popup 章节号解析（兼管文件排序）/ 正文兜底 / 去重归一化、uploader 去重归一化。

## 修法

- 前缀正则五处统一升级：`第\s*(?:\d+|[零〇一二两三四五六七八九十百千]+)\s*章`
- 新增 `cnToInt`（中文数字→整数：二十→20、三十九→39、一百零五→105、两百→200，
  非法字符 null；popup 与 publisher 各一份，MV3 无模块系统，先例同 sameTitle）
- publisher 剥前缀逻辑抽为顶层纯函数 `pureTitle`（可被源码抽取测试）
- 放弃方案：在 popup 解析时就把标题前缀剥掉（改动数据层，进度面板/CSV 报告/编辑框
  展示全部受波及，且历史会话数据口径不一致——填表层剥离影响面最小）

## 波及面与回归

- 文件排序：中文数字文件名此前按 0 排序（顺序不稳），现按章号正确排序（正向变化）
- 历史脏数据去重：已上传的"第2章 第二章 xx"式标题，归一化各剥一层后仍判同章，不误发
- uploader 番茄侧列表提号保持阿拉伯（番茄前缀由章节号框生成，恒为阿拉伯，非同一根因）
- 回归：tests/logic.test.cjs 70/70（新增 17 条红→绿，存量 53 条基线无退化）；
  `bash tests/run.sh` 通过（UI 测试本环境无 playwright 自动跳过）
