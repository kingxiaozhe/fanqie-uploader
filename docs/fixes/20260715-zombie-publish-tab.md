# 僵尸发布标签页：批次结束后残留脚本每 4 分钟点「下一步」

- 日期：2026-07-15　状态：已修复
- 修前证据：运行日志 `21:24:51 → 21:28:51 → 21:32:51 → 21:36:51` 精确 4 分钟间隔的
  「🔁 未见弹窗，重试点击下一步…」，此时批次早已结束/停止。

## 现象

某章"下一步"点击未生效、看门狗 5 分钟超时把该章判失败并推进批次后，那个发布标签页
没有被关闭，脚本残留在真实页面上持续点按钮，有误操作风险。

## 根因

`submitAndConfirm` 用 **tick 计数**当超时（`setInterval(…,1000)` + `n>=60`≈60 秒）。
Chrome 对隐藏超过 5 分钟的标签页实施 intensive throttling，定时器降到每分钟 1 次：
60 秒的预算被拉长到约 1 小时，`n%4===0` 的重点下一步恰好变成每 4 分钟一次（与日志吻合）。
发布页由 `active:false` 打开，天然是隐藏标签页。看门狗超时路径上也没有任何人负责关页
（正常路径由 TASK_DONE/TASK_FAILED 触发关页，但卡住的页什么都没发出去）。

## 修法（双闸）

1. publisher：超时改**墙钟**——`Date.now()-startMs-frozenMs >= SUBMIT_CONFIRM_TIMEOUT_MS(60s)`；
   风险检测等待用 `frozenMs += sinceLastTick` 冻结（原 `n--` 语义的墙钟版，节流下同样正确）。
   到点 resolve(false) 走既有失败路径：TASK_FAILED + 自发 CLOSE_TAB 关页。
2. uploader 看门狗触发时发 `CLOSE_PUBLISH_TAB`，background 用 `last_publish_tab_id`
  （OPEN_PUBLISH_TAB 时落 storage）强制关当前发布页——脚本已死/被节流卡死也能收口。
   守卫：dryRun 试填页不关（留给用户检查）；风控暂停已 clearWatchdog，验证页不受影响。
- 放弃方案：只加看门狗关页不改墙钟（页在被关前仍会残留点击最多 5 分钟，治标不彻底）。

## 波及面与回归

- 正常成功/失败/停止路径的关页时序不变；风控 PAUSE_BATCH 保留页面的行为不变。
- 行为变化：看门狗超时后发布页被强制关闭——修复前该页可能在"已判失败"后仍完成发布造成
  状态错位，修复后更确定；已创建草稿仍由防重复查重兜底。
- 回归：53/53 通过，基线无退化。
- Chrome 节流无法在 node 测试复现（需真实浏览器隐藏标签页>5min），修前证据=日志间隔，
  修后验证=墙钟算式直读 + 审查核验；后续真实批次跑一轮观察日志确认无残留点击。
