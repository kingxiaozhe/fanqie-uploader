- **严重 — ZIP64/解压炸弹未正确拒绝。** 输入一个解压后 5 GiB、压缩后不足 4 GiB 的 DEFLATE 条目：中央目录 `uncompressedSize=0xffffffff`，但 `compressedSize/localOffset` 为普通值。`readCentralDir` 不检查未压缩大小，随后 `Response.arrayBuffer()` 尝试完整分配数 GiB，导致弹窗或浏览器进程内存耗尽，而不是报“暂不支持 zip64”。

- **严重 — 过滤发生在解压之后。** ZIP 含正常 `第1章.txt`，另含压缩率极高的 `__MACOSX/._x.txt` 或 `.hidden.txt`。后者虽最终应被过滤，`parseZip` 却会先完整解压，仍可造成内存耗尽。类似地，只要无关的 `cover.jpg` 使用 method 93，整个导入就报“不支持的压缩方式”，合法章节也无法导入。

- **高 — 不校验 CRC，会静默导入损坏内容。** 对 STORED 的 `第1章.txt` 修改一个正文数据字节而保留中央目录 CRC 不变，`sliceEntryData` 和 `inflate` 会直接接受，最终生成内容被篡改的章节，而不是报告 ZIP 损坏。

- **中 — EOCD 可被合法注释中的伪签名误命中。** 一个正常 ZIP 的 22 字节注释本身构造成 `PK\x05\x06`，伪记录的 `commentLen=0、cdCount=0、cdOffset=0`。倒扫会先接受注释末尾的伪 EOCD，结果显示“ZIP 内没有 .txt / .md 文件”，不会继续找到真正 EOCD。

- **中 — 拼接型/SFX ZIP 偏移计算错误。** 给普通 ZIP 前置一段可执行 stub、但保持 ZIP 内部相对偏移不变（常见自解压/concatenated ZIP），`cdOffset` 需要加前置数据长度；当前代码直接从文件起点使用它，因此在 stub 或局部数据中寻找中央目录并报“ZIP 中央目录损坏”。

- **低 — 隐藏目录内容没有过滤。** 合法条目 `.draft/第1章.txt` 的 basename 是 `第1章.txt`，会被当作章节导入；过滤只检查最后一个路径分量是否以点开头。

- **低 — `BIT3` 测试常量不是有效 ZIP。** `hello` 的 CRC32 应为 `0x3610a686`，但数据描述符和中央目录都写成了 0。该损坏输入仍被测试接受，因此不能证明真实 bit3 ZIP 的完整兼容性，也掩盖了生产代码不校验 CRC 的问题。
